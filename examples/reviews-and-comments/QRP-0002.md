## 1. Summary

QRP-0002 amends QRP-0001 with seven additions and one
clarification, all aligned with the strategic concepts
introduced in
[`UseCases/trust-weighted-reviews/`](../../UseCases/trust-weighted-reviews/):

1. **DISCLOSURE** event type for sponsorship and affiliate
   disclosure.
2. **Mandatory `disclosure` field** in REVIEW payload (no
   default; must be explicit).
3. **Validation-tier resolution convention** for widgets
   rendering reviewer credentials from QDP-0023 DNS-anchored
   TRUST edges.
4. **ERASE_REQUEST** event type, resolving the TBD in QRP-0001
   §6.
5. **PROFILE_DECLARATION** event type for credentialed
   reviewers.
6. **TIP** event type for direct reviewer monetization.
7. **BRAND_BID** and **BID_ACCEPT** event types for the
   brand-disclosure marketplace.
8. **`reviews.moderation.*` subdomain convention** for
   auditor / fraud-hunter trust delegation.
9. **PURCHASE multi-currency** support (ISO 4217).
10. **Cross-network federation discount handling**
    (refers to QDP-0013 §4).

All additions are soft-fork-compatible: clients that do not
recognize the new event types ignore them safely, and the
mandatory-disclosure modification is enforced at the
presentation layer first, then at the protocol layer at
QRP-0002 activation block.

## 2. Motivation

The QRP-0001 spec captured the core wire format for
trust-weighted reviews. Eighteen months of strategic and
commercial work since then have surfaced concrete needs the
spec does not yet meet:

- **Sponsorship disclosure** is a real legal requirement
  (FTC guidance) and a real product surface (the design
  brief's S3 wireframe shows it as always-present). Today it
  lives in unstructured prose; clients render it
  inconsistently.
- **DNS-anchored validation** (QDP-0023) is the commercial
  product Quidnug LLC sells. Widgets need to know how to
  resolve a reviewer's validation tier from the operator-
  issued TRUST edges. The lookup convention is not specified.
- **Erasure requests** were marked TBD in QRP-0001 §6. EU
  Business-tier customers ask for a documented compliance
  path before signup.
- **Pro-reviewer monetization** (tipping, brand-disclosure
  marketplace, paid subscriptions) is a strategic pillar that
  has no on-protocol primitive to attach payment metadata to.
- **Auditor / fraud-hunter** is a strategic role we want to
  enable. The `reviews.moderation.*` subdomain is referenced
  in the threat model but unspecified.

Each of these is a small protocol extension. Together they
unlock the commercial launch of paid validation and the
ecosystem of professional reviewing.

## 3. Goals and non-goals

**Goals:**

- Specify each new event type's wire format, validation rules,
  stream placement, and domain.
- Make sponsorship disclosure mandatory and machine-readable.
- Define the lookup convention for validation tiers so
  widgets render consistently across implementations.
- Resolve the ERASE_REQUEST TBD with concrete operator-level
  semantics.
- Stay soft-fork-compatible with QRP-0001 deployments.

**Non-goals:**

- New rating algorithms (covered separately by extending
  `algorithm.py`/`algorithm.md`).
- New visual primitives (covered by the design brief and the
  designer's response).
- Changes to QRP-0001's existing event types beyond the
  mandatory disclosure field on REVIEW.
- Federation governance (covered by QDP-0013).

## 4. Background

QRP-0001 (`examples/reviews-and-comments/PROTOCOL.md`)
defines REVIEW, HELPFUL_VOTE, UNHELPFUL_VOTE, REPLY, FLAG,
PURCHASE, and TRUST_TOPIC. The wire formats in §5 and the
identity bootstrap in §8 are unchanged by QRP-0002. The
schema-org integration in §11 will gain new mappings.

QDP-0012 (Domain Governance) defines validators and
governors per domain. QDP-0023 (DNS-Anchored Identity
Attestation) defines how a domain is bound to a quid via
operator-issued TRUST edges under
`operators.<fqdn>.network.quidnug.com`.

## 5. Design

### 5.1 DISCLOSURE event type

**Published by:** any reviewer.
**Stream:** the reviewer's stream (NOT the product's).
**Domain:** the same domain as the review(s) it covers, or
`reviews.public.disclosures.global` for blanket disclosures.

```json
{
  "type": "EVENT",
  "subjectId": "<reviewer-quid>",
  "subjectType": "QUID",
  "eventType": "DISCLOSURE",
  "payload": {
    "qrpVersion": 2,
    "scope": "review|blanket|topic",
    "reviewTxId": "<tx-id>",
    "topicDomain": "reviews.public.technology.cameras",
    "validFrom": 1700000000,
    "validUntil": null,
    "category": "sponsored|comped|affiliate|employer|family|none",
    "sponsorQuid": "<sponsor-quid>",
    "sponsorDisplay": "AcmeBrand",
    "compensation": {
      "kind": "cash|product|service|other",
      "valueUsd": 200.00,
      "details": "$200 paid via Stripe Connect"
    },
    "termsCid": "bafy...",
    "termsHash": "sha256:..."
  }
}
```

**Field rules:**

- `scope` MUST be one of the three values. `review` covers a
  single REVIEW (referenced via `reviewTxId`); `blanket`
  applies to all subsequent reviews until `validUntil` or a
  later DISCLOSURE supersedes; `topic` applies to all
  reviews under `topicDomain` until superseded.
- `category` MUST be one of the six values. `none` is the
  explicit "no disclosure required" assertion (used by the
  blanket scope to declare a clean period).
- `sponsorQuid` MUST be present unless `category=none`.
- `compensation.valueUsd` SHOULD be present for
  `category=sponsored`, MAY be omitted for `comped` (when
  amount is genuinely unknown).
- `termsCid` and `termsHash` are optional; when present, point
  to a signed terms document (e.g., the brand-disclosure
  marketplace bid acceptance).

**Validation:**

- Verify reviewer signature.
- If `scope=review`, verify the referenced `reviewTxId`
  exists and was authored by the same reviewer.
- If `category in {sponsored, comped, affiliate}` and a
  BID_ACCEPT event references this DISCLOSURE, verify the
  bid chain (see §5.7).

**Backward compatibility:** consumers that do not recognize
DISCLOSURE events ignore them and fall back to the
mandatory `disclosure` field in REVIEW (§5.2).

### 5.2 Mandatory `disclosure` field in REVIEW

QRP-0001 §5.1 REVIEW payload is amended:

```json
{
  "qrpVersion": 2,
  "rating": 4.5,
  "maxRating": 5.0,
  "title": "...",
  "bodyMarkdown": "...",
  "locale": "en-US",
  "mediaAttachments": [],
  "contextTags": [],
  "purchaseAttestationCid": null,
  "supersedes": null,
  "disclosure": {
    "category": "none|sponsored|comped|affiliate|employer|family",
    "disclosureTxId": "<tx-id-of-DISCLOSURE-event-if-not-none>"
  }
}
```

**Rules:**

- `disclosure` is REQUIRED. A REVIEW payload without it is
  invalid at QRP-0002 activation block.
- `disclosure.category` MUST be one of the six values. No
  default. Compose UIs MUST force an explicit choice.
- If `category != none`, `disclosureTxId` MUST point to a
  DISCLOSURE event by the same reviewer with matching
  scope.
- If `category == none`, `disclosureTxId` is null.

**Backward compatibility:**

- During the transition (`qrpVersion: 1` REVIEWs published
  pre-activation), consumers SHOULD treat missing
  `disclosure` as `category: "none"` for rendering, with a
  small UI hint "disclosure not declared (legacy review)".
- Post-activation, validators reject REVIEWs with missing
  `disclosure`. The activation block is set per the
  governance process (QDP-0012).

### 5.3 Validation-tier resolution

This is a **convention**, not a new event type. Specifies how
widgets resolve a reviewer's or site's validation tier from
operator-issued TRUST edges.

For a reviewer or site quid `Q` claiming domain `D`:

1. Compute the resolution domain:
   `operators.<D>.network.quidnug.com`.
2. Query the network for the highest-weight TRUST edge with:
   - `truster` in the set of recognized validation operators
     (initially Quidnug LLC's `validation-operator-quid`;
     federated networks add their own).
   - `trustee == Q`.
   - `domain == operators.<D>.network.quidnug.com`.
3. Read the edge's `attributes` block for `tier` (`free` /
   `pro` / `business` / `partner`), `signedAt`, `kybStatus`,
   and any other operator-published metadata.
4. Render the corresponding badge.

**Tier mapping (default):**

| Tier      | Trust level | Display                |
|-----------|-------------|------------------------|
| free      | 0.2         | "Email-verified"       |
| pro       | 0.8         | "Pro tier"             |
| business  | 0.95        | "Business tier · KYB"  |
| partner   | 1.0         | "Partner"              |

**Recognized validation operators:**

The list of recognized operators is itself a TRUST edge: any
quid that the network's seed governor has trusted at level
≥ 0.95 in domain `validators.network.quidnug.com` is a
recognized validation operator. This makes the operator set
discoverable on-network rather than hardcoded in clients.

**Failure modes:**

- No matching edge found: render reviewer with no validation
  badge. Do not show "unverified" prominently; absence of
  badge is the default state.
- Edge present but expired (`validUntil` past): render with a
  faded badge and a tooltip "validation lapsed".
- Edge present but at level 0.0 (revocation): render with a
  warning badge and a "validation revoked" tooltip; observers
  may choose to suppress or downweight.

### 5.4 ERASE_REQUEST event type

Resolves QRP-0001 §6 TBD.

**Published by:** the data subject (their own quid) for
self-erasure, OR a recognized legal operator (court order,
DPA decision) for third-party erasure.

**Stream:** the subject's own stream.
**Domain:** `reviews.public.privacy.erasure` (a reserved
sibling of `reviews.public`).

```json
{
  "type": "EVENT",
  "subjectId": "<subject-quid>",
  "subjectType": "QUID",
  "eventType": "ERASE_REQUEST",
  "payload": {
    "qrpVersion": 2,
    "scope": "all|topic|event",
    "topicDomain": "reviews.public.restaurants",
    "targetTxIds": ["<tx-id-1>", "<tx-id-2>"],
    "reason": "self-request|gdpr-erasure|defamation-ruling|dmca|csam|other",
    "supportingCid": "bafy...",
    "supportingHash": "sha256:...",
    "issuedBy": "<issuer-quid>",
    "issuedAt": 1700000000
  }
}
```

**Semantics:**

- ERASE_REQUEST is **advisory**, not mandatory. The append-
  only protocol cannot delete the underlying events. What it
  does: instructs operators to **stop gossiping** the
  referenced events through their nodes.
- Operator behavior is per-operator policy; QRP-0002 specifies
  the **request format**, not the operator's response.
- An operator that honors the request publishes a separate
  `ERASE_HONORED` event (operator-stream) acknowledging which
  events they are no longer serving and the legal basis.
- Observers querying via an honoring operator stop seeing the
  events; the events remain on operators that do not honor
  the request.
- Self-erasure (subject is the issuer) SHOULD be honored by
  default by operators in jurisdictions with GDPR-equivalent
  rights (EU, UK, Brazil, California).
- Third-party erasure (court order, DPA) requires
  `supportingCid` pointing to a signed legal document; the
  operator audits before honoring.

**Validation:**

- Verify issuer signature.
- If `scope=event`, verify each target tx exists.
- If `scope=all`, no target validation needed; operator
  applies to all events on the subject's streams.
- Operators MAY require additional out-of-band verification
  before honoring; QRP-0002 does not constrain this.

### 5.5 PROFILE_DECLARATION event type

For credentialed reviewers who want to publish verifiable
expertise.

**Published by:** the reviewer.
**Stream:** the reviewer's stream.
**Domain:** the topic the credential applies to, or a parent
domain if applicable across multiple topics.

```json
{
  "type": "EVENT",
  "subjectId": "<reviewer-quid>",
  "subjectType": "QUID",
  "eventType": "PROFILE_DECLARATION",
  "payload": {
    "qrpVersion": 2,
    "topicDomain": "reviews.public.medical-devices",
    "credentialKind": "professional-license|degree|certification|publication|employer|other",
    "credentialIssuer": "American Board of Internal Medicine",
    "credentialIssuerQuid": "<issuer-quid-if-on-network>",
    "credentialIdentifier": "ABIM-cert-2019",
    "credentialAttachmentCid": "bafy...",
    "credentialAttachmentHash": "sha256:...",
    "validFrom": 1577836800,
    "validUntil": 1893456000,
    "summary": "Board-certified internist; 14 years practice."
  }
}
```

**Verification:**

- The credential is a signed payload; verification of its
  truthfulness is out-of-scope for the protocol. PROFILE_
  DECLARATION asserts the reviewer's claim; observers may
  choose to additionally trust an issuer-quid that
  cross-signs the same credential via a separate event.
- A credential's `credentialIssuerQuid` referencing an
  on-network issuer enables cryptographic verification: if
  the issuer publishes their own `CREDENTIAL_ATTESTATION`
  event referencing the same `credentialIdentifier`, the
  binding is verified end-to-end.

**Use in algorithm:**

- The rating algorithm MAY use PROFILE_DECLARATION as a
  topical-trust boost for observers who trust the credential
  issuer. Specifics in `algorithm.md` revision.

### 5.6 TIP event type

Direct reviewer monetization.

**Published by:** the tipper.
**Stream:** the tipper's stream.
**Domain:** `reviews.public.tips` (sibling of
`reviews.public`).

```json
{
  "type": "EVENT",
  "subjectId": "<tipper-quid>",
  "subjectType": "QUID",
  "eventType": "TIP",
  "payload": {
    "qrpVersion": 2,
    "recipientQuid": "<reviewer-quid>",
    "reviewTxId": "<tx-id-of-tipped-review>",
    "amount": {
      "value": 5.00,
      "currency": "USD"
    },
    "method": "stripe-connect|lightning|other",
    "methodReference": "pi_3KQ...",
    "message": "Great review, thanks."
  }
}
```

**Privacy:**

- TIP events are public by default (appear on the tipper's
  stream). Tippers wanting privacy SHOULD use the
  `recipientQuid` only and omit `reviewTxId`.
- Amount is exact; not a hash. Privacy-conscious tippers can
  publish `value: 0` and use an off-chain payment with
  `methodReference` linking to the actual receipt.
- Anonymous tipping: the tipper signs with an ephemeral quid
  that has no other on-network history. The reviewer can
  display "anonymous tip" without linking to the tipper's
  primary identity.

**Validation:**

- The protocol does not verify that the payment actually
  occurred. Tipping is a **claim** of payment; receipts come
  from the payment provider out-of-band.
- A reviewer MAY publish a corroborating
  `TIP_ACKNOWLEDGED` event linking back to the TIP
  transaction; this strengthens the claim but is not
  required.

### 5.7 BRAND_BID and BID_ACCEPT event types

The brand-disclosure marketplace, on-protocol.

**BRAND_BID published by:** a brand quid (typically a
validated site quid representing the brand).
**Stream:** the brand's stream.
**Domain:** `reviews.public.bids.<topic>` (e.g.,
`reviews.public.bids.reviews.public.technology.cameras`),
mirroring the topic of the requested review.

```json
{
  "type": "EVENT",
  "subjectId": "<brand-quid>",
  "subjectType": "QUID",
  "eventType": "BRAND_BID",
  "payload": {
    "qrpVersion": 2,
    "productAssetQuid": "<product-title-quid>",
    "topicDomain": "reviews.public.technology.cameras",
    "compensation": {
      "kind": "cash|product|service",
      "valueUsd": 200.00,
      "details": "$200 paid in 30 days via Stripe Connect"
    },
    "reviewerQualifications": {
      "minTopicTrust": 0.7,
      "minReviewsInTopic": 50,
      "minHelpfulnessRatio": 0.7,
      "validationTierRequired": "pro|business|partner|null"
    },
    "termsCid": "bafy...",
    "termsHash": "sha256:...",
    "validUntil": 1700000000,
    "maxAcceptances": 10
  }
}
```

**BID_ACCEPT published by:** a reviewer accepting the bid.
**Stream:** the reviewer's stream.
**Domain:** same as the bid's domain.

```json
{
  "type": "EVENT",
  "subjectId": "<reviewer-quid>",
  "subjectType": "QUID",
  "eventType": "BID_ACCEPT",
  "payload": {
    "qrpVersion": 2,
    "bidTxId": "<brand-bid-tx-id>",
    "expectedReviewBy": 1700000000,
    "termsAcceptedCid": "bafy..."
  }
}
```

After acceptance, the reviewer publishes their REVIEW with
`disclosure.category = "sponsored"` and a DISCLOSURE event
referencing the BID_ACCEPT (via `termsCid` or
`compensation.details`). The chain
BRAND_BID → BID_ACCEPT → DISCLOSURE → REVIEW provides
end-to-end auditable provenance for paid reviews.

**Validation:**

- BRAND_BID validity: `validUntil` not past, brand quid signed.
- BID_ACCEPT validity: bid exists, bid not expired, bid not
  oversubscribed (count active BID_ACCEPTs, compare to
  `maxAcceptances`), reviewer meets `reviewerQualifications`
  at acceptance time.
- DISCLOSURE referencing the bid: `termsCid` or another field
  ties the chain together.
- REVIEW: `disclosure.category = "sponsored"` and
  `disclosureTxId` points to the linked DISCLOSURE event.

### 5.8 `reviews.moderation.*` subdomain spec

Auditors / fraud-hunters delegate moderation authority via
TRUST_TOPIC events scoped to the
`reviews.moderation.*` subdomain tree.

```
reviews.moderation                              (root)
├── reviews.moderation.global                   (any topic)
├── reviews.moderation.technology               (topic-bound)
├── reviews.moderation.restaurants              (topic-bound)
├── reviews.moderation.spam                     (specific abuse)
├── reviews.moderation.brushing                 (specific abuse)
└── reviews.moderation.ai-generated             (specific abuse)
```

Observers wanting to delegate moderation publish:

```
TRUST_TOPIC:
  truster: <observer-quid>
  trustee: <moderator-quid>
  domain: reviews.moderation.<scope>
  trustLevel: 0.0-1.0
  description: "trusted moderator for spam in tech reviews"
  validUntil: optional
```

When the moderator publishes a FLAG event (per QRP-0001 §5.4),
the observer's rating algorithm:

1. Looks up the moderator's trust from the observer in
   `reviews.moderation.<topic-or-global>`.
2. If trust ≥ threshold (default 0.5), apply the flag's
   severity-weighted penalty to the flagged review's weight.

This makes auditor designation a first-class on-network
concept rather than a private app-level convention.

### 5.9 PURCHASE multi-currency

QRP-0001 §5.5 PURCHASE payload is amended:

```json
{
  "retailerAttestation": {
    "retailerName": "Example Retailer",
    "orderIdHash": "sha256:...",
    "amount": {
      "value": 1899.00,
      "currency": "USD"
    }
  }
}
```

Replaces `amountUsd` (deprecated but still recognized for
backward compatibility). `currency` is an ISO 4217 code.

### 5.10 Cross-network federation discount

When QDP-0013 federation is active, widgets and algorithms
encounter reviewers whose primary network is federated
rather than native. QRP-0002 specifies:

- Cross-network reviewers' contributions are weighted at the
  federation discount factor `D` (default 0.8 per cross-
  network hop), as defined in QDP-0013 §4.
- The discount is applied multiplicatively to factor `T`
  (topical trust) in the rating algorithm.
- Widgets SHOULD render a small "via FederatedNetwork.com"
  annotation on cross-network reviewer cards so observers
  understand the provenance.

## 6. Security considerations

**DISCLOSURE event tampering:** A reviewer cannot retroactively
delete a DISCLOSURE event (append-only). They can supersede it
with a contradictory DISCLOSURE; observers should distrust
reviewers with multiple contradictory disclosures on the same
review.

**ERASE_REQUEST abuse:** A bad-faith ERASE_REQUEST (e.g., to
suppress legitimate negative reviews) is identifiable: the
`reason` is wire-visible, the `supportingCid` can be audited,
operators publish their honoring decisions. Operators that
honor abusive requests damage their own credibility.

**TIP event spoofing:** A bad actor could publish fake TIP
events claiming to have tipped popular reviewers, hoping the
reviewer's auto-acknowledgement adds credibility. Mitigation:
TIP events without verifiable payment receipts should be
visually de-emphasized; reviewers should only acknowledge
TIPs they have verified.

**BID_ACCEPT race conditions:** A popular bid may receive
more BID_ACCEPTs than `maxAcceptances` before all reviewers
see the count. Resolution: validators apply first-N-by-block-
height; later acceptances are rejected with
`BID_OVERSUBSCRIBED`. Brands should set `maxAcceptances`
generously and budget accordingly.

**Validation-tier resolution attacks:** A malicious actor
publishes spoof TRUST edges from a fake "validation operator"
quid. Mitigation: clients only respect operators in the
recognized-operator set (those with TRUST edges from seed
governors at level ≥ 0.95 in `validators.network.quidnug.com`).
Any validation operator's compromise is contained.

## 7. Privacy considerations

**DISCLOSURE events are public by design.** Sponsorship
disclosure is a public good; the event is intentionally
visible. Reviewers in jurisdictions with strict employment-
disclosure laws (some EU member states, Australia, Canada)
should consult counsel before publishing employer-related
disclosures.

**TIP events expose tipper-recipient relationships.**
Privacy-conscious tippers should use ephemeral quids or
omit `reviewTxId`. The protocol gives the tools; the
practice is on the user.

**ERASE_REQUEST is not privacy by default.** Honoring an
erasure request makes content unreachable through the
honoring operator, but the event itself (the request) is
still on-chain. Subjects requesting erasure should
understand they are not deleting evidence of their request.

**PROFILE_DECLARATION exposes credentials.** Reviewers
publishing professional credentials are tying their
real-world identity to their reviewer quid. This is
intentional for pro reviewers and undesirable for
pseudonymous reviewers. Compose UIs SHOULD warn before first
publication.

## 8. Backward compatibility

| Amendment             | Compatibility class                    |
|-----------------------|----------------------------------------|
| DISCLOSURE event      | Soft-fork: new event type, ignored by old clients |
| Mandatory disclosure  | Hard-fork at activation block; reject REVIEWs without `disclosure` field |
| Validation-tier res   | Convention only; no wire change        |
| ERASE_REQUEST         | Soft-fork: new event type              |
| PROFILE_DECLARATION   | Soft-fork: new event type              |
| TIP                   | Soft-fork: new event type              |
| BRAND_BID / BID_ACCEPT| Soft-fork: new event types             |
| Moderation subdomain  | Convention only; uses existing TRUST_TOPIC |
| PURCHASE multi-currency | Soft-fork: `amount` field added; `amountUsd` retained for compat; deprecation in QRP-0003 |
| Federation discount   | Convention only; algorithm-side change  |

**Activation strategy:** the only hard-fork piece is the
mandatory `disclosure` field on REVIEW. Activation block:
6 months after QRP-0002 acceptance, allowing widget vendors
and SDKs time to update their compose flows.

## 9. Test vectors

### 9.1 DISCLOSURE event for a sponsored review

Input:
```json
{
  "scope": "review",
  "reviewTxId": "tx_review_001",
  "category": "sponsored",
  "sponsorQuid": "quid_acmebrand",
  "sponsorDisplay": "AcmeBrand",
  "compensation": {
    "kind": "cash",
    "valueUsd": 200.00,
    "details": "$200 via Stripe Connect"
  }
}
```

Expected: DISCLOSURE event accepted; the matching REVIEW must
have `disclosure.category = "sponsored"` and
`disclosure.disclosureTxId` referencing this DISCLOSURE.

### 9.2 REVIEW with mandatory disclosure (post-activation)

Valid:
```json
{
  "qrpVersion": 2,
  "rating": 4.5,
  "maxRating": 5.0,
  "bodyMarkdown": "...",
  "disclosure": { "category": "none", "disclosureTxId": null }
}
```

Invalid (missing disclosure):
```json
{
  "qrpVersion": 2,
  "rating": 4.5,
  "maxRating": 5.0,
  "bodyMarkdown": "..."
}
```
Expected: validation error
`MISSING_REQUIRED_FIELD: disclosure`.

### 9.3 BRAND_BID → BID_ACCEPT → DISCLOSURE → REVIEW chain

1. Brand publishes BRAND_BID at tx `tx_bid_001`.
2. Reviewer publishes BID_ACCEPT at tx `tx_acc_001`
   referencing `tx_bid_001`.
3. Reviewer publishes DISCLOSURE at tx `tx_disc_001` with
   `compensation.details` referencing `tx_acc_001`.
4. Reviewer publishes REVIEW at tx `tx_rev_001` with
   `disclosure.category = "sponsored"` and
   `disclosure.disclosureTxId = "tx_disc_001"`.

Expected: full chain validates; observers can audit the
sponsored relationship.

### 9.4 Validation-tier resolution

For reviewer quid `Q` claiming `alice-eats.com`:

```
GET /api/trust?
    truster=<validation-operator>&
    trustee=Q&
    domain=operators.alice-eats.com.network.quidnug.com
```

Response:
```json
{
  "trustLevel": 0.95,
  "attributes": {
    "tier": "business",
    "kybStatus": "approved",
    "signedAt": "2026-04-12T10:00:00Z"
  }
}
```

Widget renders: "alice-eats.com ✓ Business · KYB verified".

## 11. Open questions

1. **PROFILE_DECLARATION verification.** Should the protocol
   require an issuer counter-signature for credential claims
   to be considered "verified," or leave verification entirely
   to observers' trust graphs? (Current: leave to observers.)

2. **TIP event amount visibility.** Should privacy-preserving
   default be amount-hidden with a separate signed receipt,
   or amount-visible by default? (Current: visible.)

3. **BRAND_BID matchmaking.** Should the protocol specify a
   matchmaking algorithm (sort bids by reviewer rep), or leave
   to client-side discovery? (Current: client-side; future
   QRP may add a discovery service.)

4. **Cross-network DISCLOSURE.** When a sponsored review
   crosses federation hops, do the federation discount and
   the disclosure visibility interact? (Open; needs QDP-0013
   coordination.)

5. **Erasure granularity.** Should ERASE_REQUEST support
   field-level erasure (e.g., redact author name but keep
   review body) in addition to event-level? (Current:
   event-level only; field-level is a future amendment.)
